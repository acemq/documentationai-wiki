# Authentication & Authorization

## Built-in authentication backends

| Backend | Description |
|---|---|
| `rabbit_auth_backend_internal` | Username/password in broker database (default) |
| `rabbit_auth_backend_ldap` | LDAP/Active Directory |
| `rabbit_auth_backend_oauth2` | OAuth 2.0 / JWT tokens |
| `rabbit_auth_backend_cache` | Caches results from another backend |

## LDAP integration

```ini
# rabbitmq.conf
auth_backends.1 = ldap
auth_backends.2 = internal

auth_ldap.servers.1 = ldap.company.internal
auth_ldap.port = 636
auth_ldap.use_ssl = true
auth_ldap.user_dn_pattern = cn=${username},ou=Services,dc=company,dc=internal
auth_ldap.vhost_access_query.type = constant
auth_ldap.vhost_access_query.value = true
auth_ldap.resource_access_query.type = constant
auth_ldap.resource_access_query.value = true
```

## Password hashing

RabbitMQ stores passwords as salted SHA-256 or SHA-512 hashes:

```bash
# Set password (broker hashes it automatically)
rabbitmqctl add_user app-svc "$(openssl rand -base64 32)"

# Or use the management API
curl -u admin:pass -X PUT \
  -H 'Content-Type: application/json' \
  -d '{"password":"newpass","tags":""}' \
  https://rabbitmq.internal:15671/api/users/app-svc
```

## Scoping permissions by topic

For MQTT and STOMP, topic-level authorization:

```bash
# Allow publishing only to sensors/* topics
rabbitmqctl set_topic_permissions -p mqtt sensor-device \
  amq.topic "^sensors\." "^sensors\."
```

## Service account rotation procedure

```bash
# 1. Create new user with same permissions
rabbitmqctl add_user app-svc-v2 "$(vault read -field=password secret/rabbitmq/app-svc)"
rabbitmqctl set_permissions -p production app-svc-v2 "" ".*" ".*"

# 2. Roll out new credentials to applications
# 3. Verify connections from new user exist
rabbitmqctl list_connections user | grep app-svc-v2

# 4. Remove old user
rabbitmqctl delete_user app-svc
```

## AceMQ Take

- Delete the `guest` user in production or change its password. It defaults to `guest`/`guest` with full access.
- Use the `cache` backend in front of LDAP to reduce LDAP query overhead on high-connection workloads.
- OAuth 2.0 (JWT) authentication is the best choice for Kubernetes workloads where pods receive OIDC tokens from the cluster identity provider.
