🐇 **Upstream reference:** [MQTT Plugin](https://www.rabbitmq.com/docs/mqtt)

The `rabbitmq_mqtt` plugin enables MQTT protocol support. RabbitMQ supports MQTT 3.1, 3.1.1, and 5.0. MQTT is ideal for IoT devices, mobile clients, and constrained environments.

## Enabling MQTT

```bash
rabbitmq-plugins enable rabbitmq_mqtt
```

Default ports: `1883` (cleartext), `8883` (TLS).

## MQTT and AMQP Mapping

MQTT topics map to AMQP routing keys with `/` replaced by `.`:

| MQTT topic | AMQP routing key |
|---|---|
| `sensors/temperature/kitchen` | `sensors.temperature.kitchen` |
| `devices/+/status` | `devices.*.status` (wildcard) |
| `events/#` | `events.#` (multi-level wildcard) |

MQTT messages are published to the `amq.topic` exchange by default. MQTT subscriptions create queues bound to `amq.topic` with the corresponding AMQP routing key.

## QoS Levels

| QoS | AMQP Equivalent | Behavior |
|---|---|---|
| QoS 0 | No confirms | Fire-and-forget; no delivery guarantee |
| QoS 1 | Publisher confirms | At-least-once; duplicate possible |
| QoS 2 | Two-phase delivery (MQTT 5.0) | Exactly-once within MQTT session |

## Configuration

```ini
mqtt.listeners.tcp.default = 1883
mqtt.listeners.ssl.default = 8883
mqtt.vhost = /
mqtt.default_user = mqtt-user
mqtt.default_pass = s3cr3t
mqtt.allow_anonymous = false
mqtt.prefetch = 10
```

## Persistent Subscriptions

MQTT clients can use a `client_id` and `clean_session = false` to create a durable subscription. RabbitMQ creates a durable queue named after the client ID. Messages published while the client is offline are delivered on reconnect.

## MQTT 5.0 Features

RabbitMQ 3.11+ supports MQTT 5.0, adding:
- Enhanced authentication (AUTH packet)
- Session expiry intervals
- Message expiry
- User properties (maps to AMQP headers)
- Reason codes for improved error handling
- Subscription identifiers

## Web MQTT (WebSocket)

```bash
rabbitmq-plugins enable rabbitmq_web_mqtt
```

Browser MQTT clients connect via WebSocket on port 15675 (`/ws` path).

## AceMQ Take

**MQTT for IoT edge, AMQP for backend.** MQTT clients (sensors, devices, mobile apps) connect via MQTT; backend services connect via AMQP. RabbitMQ bridges the two — MQTT publishes arrive as AMQP messages on `amq.topic` that any AMQP consumer can receive.

**Always use QoS 1 or QoS 2 for critical telemetry.** QoS 0 is lossy by design. For sensor readings that drive alerts or billing, use QoS 1 (at-least-once) and design consumers for idempotency.

**Disable anonymous access.** `allow_anonymous = false` with specific MQTT credentials is mandatory in production. Anonymous MQTT is a security liability in any deployment that's network-accessible.
