🐇 **Upstream reference:** [Clustering Guide](https://www.rabbitmq.com/docs/clustering)

A RabbitMQ cluster is a logical grouping of nodes that share users, virtual hosts, queues, exchanges, bindings, policies, and distributed state. Nodes communicate over Erlang distribution (OTP) and form a peer mesh.

This guide covers cluster formation, node identity, data replication, and operational topics. Starting with RabbitMQ 4.3, all replicated state uses Raft (Khepri metadata store); Mnesia is removed.

## What is a Cluster?

- All nodes share **metadata** (vhosts, exchanges, bindings, users, policies) via Khepri (Raft-based)
- **Message data** is NOT replicated by default — quorum queues and streams replicate messages; classic queues don't
- Clients can connect to any node; RabbitMQ routes operations to the appropriate leader node transparently

## Node Names

Every node has a name: `rabbit@hostname` (e.g. `rabbit@rmq-node-1.internal`). Node names must be unique within a cluster. Hostnames must resolve from all other nodes.

```bash
# Check node name
rabbitmqctl status | grep node:

# Use long names for FQDN environments
export RABBITMQ_USE_LONGNAME=true
```

## Cluster Formation Options

| Method | Description |
|---|---|
| Config file `cluster_nodes` | Static list of seed nodes |
| DNS-based discovery | SRV or A record lookup |
| AWS EC2 plugin | Tag-based peer discovery |
| Kubernetes plugin | Pod label discovery |
| Consul / etcd plugins | Service registry discovery |
| `rabbitmqctl join_cluster` | Manual |

See the [Cluster Formation](https://www.rabbitmq.com/docs/cluster-formation) guide for peer discovery details.

## Erlang Cookie

All nodes in a cluster must share the same **Erlang cookie** (`~/.erlang.cookie` or `$RABBITMQ_ERLANG_COOKIE`). This is the shared secret that authenticates inter-node communication. CLI tools must also have the same cookie to connect.

```bash
# Verify cookie on a node
cat /var/lib/rabbitmq/.erlang.cookie
```

## Recommended Node Count

| Count | Tolerated Failures | Quorum Possible? |
|---|---|---|
| 1 | 0 | N/A |
| 2 | 0 | No (2-node clusters are strongly discouraged) |
| 3 | 1 | Yes |
| 5 | 2 | Yes |
| 7 | 3 | Yes |

Two-node clusters are strongly discouraged — they cannot form quorum after a single node failure.

## Replica Placement

Quorum queue and stream leaders are placed by the broker using configured strategies:
- `balanced` (default) — distributes leaders across nodes
- `client-local` — leader on the node the declaring client connected to
- `random` — random node selection

## Node Restarts and Rejoin

When a node restarts, it attempts to contact its last-known cluster peers. If peers are reachable, it rejoins and syncs missing Raft log entries. If unreachable (network partition or cluster reset), it starts as a standalone node — which can split the cluster.

Don't force-boot a node that was partitioned without understanding the data loss implications.

## Removing and Resetting Nodes

```bash
# Remove a node from the cluster (run on a different node)
rabbitmqctl forget_cluster_node rabbit@old-node

# Reset a node to standalone (run on the node itself)
rabbitmqctl stop_app
rabbitmqctl reset
rabbitmqctl start_app
```

## AceMQ Take

**Always run odd-numbered clusters.** 3-node is the minimum for production. 5-node for multi-DC or higher availability. Never run 2-node — it provides no improvement over single-node for quorum scenarios and creates a false sense of HA.

**Pin Erlang cookies in IaC.** The Erlang cookie should be a stable, pre-provisioned secret (from a secret manager), not generated by the OS. Nodes that generate their own cookie at startup will never join a cluster.

**Test node failure and rejoin regularly.** Quorum queue behavior during node failures is deterministic and well-documented — but you should verify it in your environment. Kill a node, observe leader election, restart it, observe sync. This should be a documented runbook, not a fire drill.

**Use peer discovery plugins for Kubernetes and cloud.** Manual `join_cluster` doesn't scale to dynamic infrastructure. Use the Kubernetes peer discovery plugin so pods auto-cluster on startup without operator intervention.
