# Policy Management

## Policy basics

Policies apply runtime configuration to queues, exchanges, or both based on name patterns. They are the preferred way to configure queue properties because they can be changed without redeploying applications.

```bash
# Policy format:
# rabbitmqctl set_policy <name> <pattern> <definition> [options]

# Example: set delivery-limit on all quorum queues
rabbitmqctl set_policy delivery-limit ".*" \
  '{"delivery-limit": 3}' \
  --apply-to quorum-queues \
  --priority 50
```

## Operator policies vs. user policies

| | User policy | Operator policy |
|---|---|---|
| Set by | Application teams | Ops/platform team |
| Can override | Other user policies | User policies |
| Command | `set_policy` | `set_operator_policy` |
| Use for | App-specific config | Platform-wide defaults |

```bash
# Operator policy: set max-length on all queues
rabbitmqctl set_operator_policy max-length ".*" \
  '{"max-length": 1000000}' \
  --apply-to queues
```

## Policy priority resolution

When multiple policies match a queue, the one with the highest priority wins:

```bash
# List policies in priority order
rabbitmqctl list_policies --formatter pretty_table
```

Policy priority values: higher number wins. No default priority = 0.

## Standard production policy set

```bash
# 1. Quorum queues: delivery limit
rabbitmqctl set_policy quorum-delivery-limit ".*" \
  '{"delivery-limit": 3}' \
  --apply-to quorum-queues --priority 100

# 2. All queues: max length (safety net)
rabbitmqctl set_operator_policy max-length-safety ".*" \
  '{"max-length": 5000000, "overflow": "reject-publish"}' \
  --apply-to queues --priority 1

# 3. Retry queues: 30s TTL + DLX routing back to main
rabbitmqctl set_policy retry-ttl ".*\.retry" \
  '{"message-ttl": 30000, "dead-letter-exchange": ""}' \
  --apply-to queues --priority 200

# 4. Dead-letter queues: long TTL for investigation
rabbitmqctl set_policy dead-ttl ".*\.dead" \
  '{"message-ttl": 604800000}' \
  --apply-to queues --priority 200
```

## Verify effective policies

```bash
# Which policy applies to a specific queue?
rabbitmqctl list_queues name effective_policy_definition --formatter pretty_table

# Via management API
curl -u admin:pass https://rabbitmq:15671/api/queues/%2F/orders | \
  jq '.effective_policy_definition'
```

## AceMQ Take

- Use operator policies for platform-wide defaults (max-length, delivery-limit). Application teams set user policies for their specific needs. Operator policies win when they conflict.
- Always specify `--apply-to` explicitly. The default is `all` which applies to both queues and exchanges — most policies should be queue-only.
- Export all policies to a definitions file and keep it in version control. This is your policy audit trail.
